# SA-201 — Force Update Modal

## 1. What we built (in plain English)

A **blocking modal** that appears when the user is running an outdated version
of the app and the server has flagged the update as **mandatory**. The user
cannot dismiss it — they must click **"Restart and Update Now"**, which
auto-saves any in-flight work, downloads the update via `electron-updater`,
and quits + relaunches the app on the new version.

It also surfaces real-time status (`Ready to Install` → `Downloading… N%` →
`Installing…` → `Restarting…`) and an error+retry path if the download
fails.

## 4. How it works (request → install flow)

```
[User launches app, mandatory update detected by SA-202]
     ↓
VersionChecker renders <ForceUpdateModal status="ready" .../>
     ↓
User clicks "Restart and Update Now"
     ↓
VersionChecker.handleInstall():
  1. setInstallStatus('downloading')        ← optimistic UI flip
  2. await window.api['analysis:saveAutoSave']()  ← AC-5: persist before restart
  3. await window.api.installUpdate()       ← kick off install
     ↓
main process IPC handler (src/main/ipc/index.ts):
  - if !app.isPackaged → run dev-fallback timer (demo)
  - else:
      autoUpdater.removeAllListeners(...)   ← prevent leak on retry
      autoUpdater.on('download-progress', → send UPDATE_PROGRESS + UPDATE_STATUS='downloading')
      autoUpdater.on('update-downloaded', → send UPDATE_STATUS='installing', then quitAndInstall())
      autoUpdater.on('error',             → send UPDATE_STATUS='error')
      await autoUpdater.checkForUpdates()
      await autoUpdater.downloadUpdate()
     ↓
Renderer receives status pushes via preload's onUpdateStatus / onUpdateProgress
     ↓
Modal re-renders with new status (downloading % → installing → restarting)
     ↓
[App quits and relaunches on new version]
```

## 6. Key design decisions

### a. Use Radix Dialog instead of a custom modal
- **Why:** Accessibility, focus trap, ESC/click-outside handling come for free. Doing this by hand is error-prone (screen readers, focus restoration on close, etc.).
- **Trade-off:** One more npm dep; we get a battle-tested accessibility implementation in return.

### b. Dev fallback in `INSTALL_UPDATE` handler (`if (!app.isPackaged)`)
- **Why:** `electron-updater` requires a packaged app with publish config. In `npm run dev` neither exists, so a real call would always error. The fallback simulates the download → install → restart sequence (with progress %), so the modal flow is exercisable locally without infrastructure.
- **Trade-off:** Slightly more code in the handler. Worth it — dev experience is 10× better with this in place.

### c. `autoUpdater.autoDownload = false; autoUpdater.autoInstallOnAppQuit = false`
- **Why:** We want explicit user consent before downloading and explicit transition into install. Defaults would auto-download silently after a check and silently apply on next quit — both wrong for a *user-visible* force-update flow.

### d. Listener cleanup on every install attempt (`removeAllListeners`)
- **Why:** If the user hits Retry after an error, we register `download-progress` / `update-downloaded` / `error` again. Without cleanup we'd accumulate listeners across retries (memory leak + duplicate UI events).

### e. Auto-save as best-effort (renderer swallows errors)
- **Why:** AC-5 says auto-save *before* restart, but a save failure should NOT block the security update. Wrapping the call in `try { … } catch {}` ensures the install still proceeds even if persistence is broken.
- **Trade-off:** A failed save is silent in the renderer. Main process still logs it.

### f. Versioned `AutoSavePayload` (`schemaVersion: 1`)
- **Why:** Future-proofs the autosave format. When a richer analysis-persistence story lands, the loader can route based on `schemaVersion` and migrate old files in place instead of breaking them.

# SA-202 — Version Check on Launch

## 1. What we built (in plain English)

The **brain** behind SA-201 — a controller that runs on app launch and every
30 minutes thereafter, calls the update server, decides whether to show the
blocking modal (SA-201) / a non-blocking banner / a warning toast, and
handles all the edge cases: minimum-version enforcement, server outage,
exponential backoff retries, and offline launches via a local cache.

## 2. Files involved

| File | Role |
|---|---|
| `src/renderer/components/VersionChecker.tsx` | The controller — polls the update server, manages 5 states (`idle` / `upToDate` / `optional` / `mandatory` / `failed`), drives the modal/banner/toast rendering. |
| `src/main/ipc/index.ts` | `CHECK_FOR_UPDATES` handler — fetches the update manifest, writes to disk cache on success, serves from cache on network failure. |
| `src/preload/index.ts` | Exposes `checkForUpdates`, `getAppVersion`, `getEnv` to the renderer. |
| `src/shared/types.ts` | `UpdateCheckResult`, `CachedUpdateManifest`, `AppEnv` (incl. `UPDATE_SERVER_URL`), IPC channel names. |
| `src/main/index.ts` | Boots MSW dev mock server when `NODE_ENV === 'development'`. |
| `src/main/mocks/server.ts` | MSW server instance. |
| `src/main/mocks/handlers.ts` | MSW request handler — intercepts the update-check URL and returns canned mandatory-update response with release notes. |
| `src/renderer/App.tsx` | Mounts `<VersionChecker />` so the check runs on every launch. |
| `src/renderer/__tests__/lifecycle/VersionCheckOnLaunch.test.tsx` | Controller logic tests (mocked IPC). |
| `src/renderer/__tests__/lifecycle/VersionCheckMSW.test.tsx` | HTTP-layer parsing tests (real fetch, MSW responses). |
| `README.md` | Documents the `UPDATE_SERVER_URL` env var. |

## 3. Tech stack & why

| Tech | Why chosen | Alternative considered |
|---|---|---|
| **`semver`** | Official semver-spec parser. Required for the minimum-version gate (AC-5). Rejects invalid input — see SA-201 §3. | Custom 3-part numeric compare (replaced for correctness) |
| **Node `fetch` + `process.env`** | Native, no dep, 12-factor config. The IPC handler reads `UPDATE_SERVER_URL` from env with a hardcoded dev fallback. | `axios` (adds dep, no real benefit for one GET), config file (less standard for desktop apps) |
| **MSW (Mock Service Worker)** | Industry-standard fetch-interception library. Lets the dev/test build run end-to-end without a real update server. Used by Stripe, Vercel, Netflix. | Custom fetch monkey-patch (brittle), nock (less ergonomic, deprecated for browser code), no mock + real network (dev-experience nightmare) |
| **Native `setTimeout` (self-rescheduling)** | Polls every 30 min. Self-rescheduling instead of `setInterval` lets us vary the delay (backoff) and avoid overlapping retries. Zero deps. | `setInterval` (can't vary delay), `node-cron` (overkill for one fixed interval), OS cron (wrong tool — modal can't render when app is closed) |
| **`fs/promises` JSON cache at `{userData}/update-manifest.json`** | Survives offline launches. `userData` is Electron's standard per-user app folder. Same pattern VS Code, Slack, Discord use. | Redis (network-backed — contradicts the offline-cache use case), SQLite (overkill for one ~2KB blob), `electron-store` (ergonomic wrapper, no real benefit) |
| **`dotenv` (already in deps)** | Loaded by `process.env.UPDATE_SERVER_URL` — the env var pattern matches the existing `API_BASE_URL`, `APP_STAGE` convention. | Build-time inlining (no env-time flexibility), hardcoded (gap that broke "no hardcoding" review) |
| **React FC + TypeScript + Tailwind + Vitest** | Ticket spec; codebase convention. | n/a |

## 4. How it works (full launch flow)

```
[App launches]
     ↓
src/renderer/App.tsx mounts <VersionChecker />
     ↓
VersionChecker.useEffect fires:
  - resolves localVersionPromise (window.api.getAppVersion)
  - calls runCheck() immediately
     ↓
runCheck():
  Promise.all([
    window.api.checkForUpdates(),  ← goes through preload → main IPC
    localVersionPromise,
  ])
     ↓
main process IPC handler (CHECK_FOR_UPDATES):
  - reads UPDATE_SERVER_URL from process.env (fallback: updates.example.com)
  - fetches: GET {url}?version={current}&platform={os}
  - in dev: MSW intercepts → returns SAMPLE_UPDATE_RESPONSE
  - on success: writes {result, cachedAt: now} to userData/update-manifest.json
  - on failure: tries reading cache; if fresh (≤7 days) → returns cached
                                       else → rethrows the network error
     ↓
Back in renderer's runCheck.then():
  - failures = 0 (reset backoff counter)
  - if !result.updateAvailable → setCheckState({kind: 'upToDate'})
  - else if mandatory or below minimumVersion (via semver.lt):
      → mandatoryLockedRef.current = true
      → setCheckState({kind: 'mandatory', ...})  ← triggers SA-201 modal
      → DO NOT schedule next check (modal handles install)
  - else → setCheckState({kind: 'optional', newVersion})
  - if not locked → scheduleNext(30 min)
     ↓
runCheck.catch():
  - failures += 1
  - setCheckState({kind: 'failed'})
  - scheduleNext(backoffDelay(failures))  ← 30 * 2^failures min ± 20% jitter, capped at 6h
     ↓
[Loop continues every 30 min on success / longer on failure, until app closes or mandatory modal locks]
```

## 6. Key design decisions

### a. Self-rescheduling `setTimeout` instead of `setInterval`
- **Why:** Need variable delays for exponential backoff. `setInterval` fires on a fixed schedule regardless of whether previous handler finished — would also cause overlapping retries against a slow server.
- **Trade-off:** Slightly more code, but no race conditions on cancellation and clean backoff support.

### b. Externalize URL via `UPDATE_SERVER_URL` env var
- **Why:** 12-factor config. Hardcoded URLs fail review at any reasonable engineering org. Matches existing `API_BASE_URL` pattern in this codebase. Different environments (dev/staging/prod) need different update servers.
- **Trade-off:** A hardcoded fallback `'https://updates.example.com/...'` still exists in three places (IPC handler, MSW handler, MSW test) — they must stay in sync if the dev default ever changes. We discussed centralizing this into a `src/shared/update-server.ts` constant; chose not to in favor of keeping the codebase flatter.

### c. Disk cache at `{userData}/update-manifest.json`
- **Why:** Offline launches must still apply the mandatory-update gate. `userData` is Electron-standard, writable, per-user, survives app updates. Same pattern VS Code/Slack/Discord use. Considered Redis (wrong — network-backed contradicts offline use case), SQLite (overkill for one JSON blob), `electron-store` (ergonomic but no real benefit for a single file).
- **Trade-off:** Not atomic — a crash mid-write could corrupt the file (we recover gracefully via `catch → return null`, so impact is "lose cache once, refetch on next launch"). Atomic-rename pattern is a possible follow-up if desired.

### d. 7-day cache freshness window
- **Why:** Long enough that a few days offline isn't an issue. Short enough that a stale "mandatory" gate doesn't lock users out forever after the server fixes the manifest.
- **Trade-off:** Not configurable (yet). If needed, could be moved to an env var.

### e. Exponential backoff with ±20% jitter, capped at 6h
- **Why:** Industry standard — AWS, GCP, Stripe SDKs all use this pattern. Jitter prevents "thundering herd": when the server recovers from an outage, all 100k clients don't simultaneously retry and crash it again. 6h cap means we recover within a day even after a long outage.
- **Math:** failure 1 → 60min (48–72 actual), failure 2 → 120min (96–144), failure 3 → 240min (192–288), failure 4+ → 360min cap (288–432 actual).
- **Trade-off:** A user offline for 6h+ won't notice the moment connectivity returns — next attempt could be up to ~7h away. Acceptable for non-critical polling.

### f. "Checking for updates…" indicator with 200ms delayed reveal
- **Why:** Material Design + Apple HIG both say progress indicators should not flash for sub-200ms operations (avoid "flash of loading content" anti-pattern). Most checks resolve in <100ms (MSW or healthy server), so the indicator never renders. Slow/offline checks reveal it after the threshold.
- **Trade-off:** Hard to see in dev (MSW is too fast) — manual testing requires temporarily slowing MSW.

### g. MSW for dev/test only
- **Why:** Lets `npm run dev` exercise the full lifecycle without a real update server. Same data flow works in tests via `VersionCheckMSW.test.tsx`.
- **Gated by `NODE_ENV === 'development'`** in `src/main/index.ts` so prod builds never include MSW.

### h. Server URL host = `updates.example.com` (placeholder)
- **Why:** IETF [RFC 2606](https://datatracker.ietf.org/doc/html/rfc2606) reserves `*.example.com` for documentation and code samples — it will never resolve to a real prod server. Safer than a made-up real-looking host that could accidentally hit someone else's domain.
- **Override:** Set `UPDATE_SERVER_URL` env var in prod environment when the real URL is known.

# Appendix: Shared infrastructure

These pieces aren't owned by a single story but are touched by both SA-201
and SA-202.

## IPC channel registry (`src/shared/types.ts → IPC_CHANNELS`)

Single source of truth for IPC channel names. Both main (handler
registration) and preload (`ipcRenderer.invoke` calls) reference the same
constants — eliminates typos and makes "find references" work cleanly.

## contextBridge (`src/preload/index.ts`)

Exposes a typed `window.api.*` to the renderer with no Node access (sandbox
preserved per Electron security best practices: `nodeIntegration: false`,
`contextIsolation: true`, `sandbox: true` in `src/main/index.ts`).

## Test mock for `window.api` (`src/renderer/__tests__/setup.ts`)

Default `vi.fn()` mocks for every API method so any renderer test that
mounts a component touching `window.api` works without per-test setup.
Specific tests override individual methods as needed.

## Build / bundling

`electron-vite` with `externalizeDepsPlugin` for main + preload (so
`electron-updater`, `fs`, etc. are not bundled), React + Vite for renderer.
Path aliases (`@/`, `@main/`, `@shared/`, `@preload/`) defined in
`electron.vite.config.ts`.
